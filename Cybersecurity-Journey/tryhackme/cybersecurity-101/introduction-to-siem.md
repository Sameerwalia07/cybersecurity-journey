## 🧠 In My Own Words

A **SIEM (Security Information and Event Management)** system is the core
security solution a SOC analyst works with day to day. Logs come from two
broad categories of sources:
- **Host-centric log sources** — logs generated by individual systems
  (servers, workstations)
- **Network-centric log sources** — logs generated by network
  devices/traffic

On paper, it sounds simple — collect logs, analyse them, spot malicious
activity. In practice, this is genuinely hard because of several challenges:
- **Numerous log sources** — too many systems generating logs to review
  individually
- **No centralization** — logs scattered across many separate systems
  instead of one place
- **Limited context** — a single log entry alone often doesn't tell the
  full story
- **Limited analysis** — manually reviewing raw logs at scale isn't
  realistic
- **Format issues** — different sources log data in different, inconsistent
  formats

This is exactly the gap a **SIEM** fills. A SIEM collects logs from many
different types of sources, **standardises (normalises)** their format into
one consistent structure, **correlates** them together, and uses
**detection rules** to identify malicious activity automatically.

**Core SIEM features:**
- **Centralized Log Collection** — pulling logs from across the environment
  into a single place
- **Normalization of Logs** — converting differently formatted logs into a
  consistent, comparable structure
- **Correlation of Logs** — connecting related events across different
  sources to reveal patterns a single log alone wouldn't show
- **Real-time Alerting** — flagging suspicious activity as it happens,
  rather than only during a manual review
- **Dashboards and Reporting** — visual summaries that make it faster for
  analysts to understand the current security state

**Windows** records every system event, viewable through the **Event
Viewer**, with each type of activity assigned a **unique Event ID** —
making it easy for an analyst to quickly identify and track specific kinds
of activity.

**Common Linux log locations:**
- `/var/log/httpd` — HTTP request/response and error logs
- `/var/log/cron` — events related to scheduled cron jobs
- `/var/log/auth.log` and `/var/log/secure` — authentication-related logs
- `/var/log/kern` — kernel-related events

**Log ingestion methods** — how logs actually get into a SIEM in the first
place:
- **Agent/forwarder** — software installed on a system that sends logs to
  the SIEM automatically
- **Syslog** — a standard protocol for sending log messages over the
  network
- **Manual uploading** — logs uploaded directly, rather than streamed live
- **Port forwarding** — routing log traffic to the SIEM through a specific
  network port

Finally, the room covered what's actually **behind an alert** — how a
**detection rule** gets created in the first place (defining the specific
pattern/condition that should trigger an alert), and what the **alert
investigation** process looks like once triggered.
